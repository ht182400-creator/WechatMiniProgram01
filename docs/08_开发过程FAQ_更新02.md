### FAQ-021: POST /api/backtest/run 返回 422 Unprocessable Content

**发生时间：** 2026-05-19 23:38
**问题类型：** API 请求参数验证错误
**影响范围：** 回测功能

---

#### 问题描述

前端调用 `backtestApi.run()` 发送 POST 请求时报错：

```
POST http://localhost:5173/api/backtest/run 422 (Unprocessable Content)
```

---

#### 错误分析

后端 `/api/backtest/run` 接口使用 Query 参数接收请求：

```python
@router.post("/run", summary="运行回测")
async def run_backtest(
    code: str = Query(...),
    strategy_name: str = Query(...),
    ...
)
```

而前端使用 Axios 发送 JSON 请求体，导致参数无法正确解析。

---

#### 解决方案

将 Query 参数改为 Pydantic BaseModel 接收 JSON 请求体：

```python
class BacktestRequest(BaseModel):
    """回测请求参数"""
    code: str
    strategy_name: str
    start_date: Optional[str] = None
    end_date: Optional[str] = None
    initial_cash: float = 100000
    params: Optional[List[Dict[str, float]]] = None

@router.post("/run", summary="运行回测")
async def run_backtest(request: BacktestRequest) -> Dict[str, Any]:
    ...
```

**修改文件：** `backend/api/backtest.py`

### FAQ-024: 日期显示为 "272" 而非有效日期字符串

**发生时间：** 2026-05-19 23:44
**问题类型：** DataFrame 索引问题
**影响范围：** 所有策略生成的交易信号

---

#### 问题描述

错误日志显示：
```
[ERROR] 回测执行失败: Given date string "272" not likely a datetime, at position 0
```

权益曲线图表中 X 轴显示 "272" 而非有效日期。

---

#### 错误分析

在回测引擎执行 `reset_index(drop=True)` 后，DataFrame 的索引从 datetime 类型变成了整数 (0, 1, 2...)。

代码中试图从索引获取日期：
```python
# 错误：索引此时是整数 0, 1, 2...
date_str = str(idx.date())
```

---

#### 解决方案

所有策略文件中，从 `date` 列获取日期而非依赖索引：

```python
for idx, row in df.iterrows():
    # 从 date 列获取日期，而非依赖索引
    date_val = row['date']
    if hasattr(date_val, 'strftime'):
        date_str = date_val.strftime('%Y-%m-%d')
    else:
        date_str = str(date_val)
```

### FAQ-025: 权益曲线图不显示

**发生时间：** 2026-05-19 23:46
**问题类型：** 数据稀疏
**影响范围：** 回测结果图表

---

#### 问题描述

回测结果显示了统计数据，但权益曲线图表为空白。

---

#### 错误分析

`_execute_signal` 方法仅在 `SignalType.HOLD` 时记录权益曲线，导致数据点过于稀疏。

---

#### 解决方案

修改为每次执行信号时都记录权益曲线：

```python
def _execute_signal(self, signal: TradingSignal) -> None:
    """执行交易信号"""
    # 每次都记录权益曲线（原来只在 HOLD 时记录，数据太稀疏）
    self._record_equity(signal.date)
    
    if signal.signal == SignalType.HOLD:
        return
    # ... BUY/SELL logic
```

## 技术要点总结

### 1. API 设计规范

- FastAPI 推荐使用 Pydantic BaseModel 接收 JSON 请求体，而非 Query 参数
- 保持 API 响应结构一致性，包含必要的数据字段

### 2. pandas DataFrame 索引注意事项

- `reset_index(drop=True)` 会将原索引转换为列，同时重置索引为整数
- 访问日期数据时，应从原始 `date` 列获取，而非依赖索引

### 3. 前端参数传递

- Axios 的 params 选项期望对象格式
- 避免传递原始类型（如数字）作为参数

### 4. 数据可视化

- ECharts 图表需要足够的数据点才能正常显示
- 权益曲线应在每一步都记录，而非仅在状态变化时记录

### FAQ-029: 股票详情页价格涨跌幅显示为 0.0

**发生时间：** 2026-05-20 00:05
**问题类型：** 数据字段映射错误
**影响范围：** 股票详情页

---

#### 问题描述

股票详情页涨跌幅显示为 "0.0 / 0.0%"

---

#### 错误分析

前端使用 `stockInfo.value.pct_change` 获取涨跌幅，但 tushare `realtime_quote` 返回的字段名是：
- `price`（现价）而非 `close`
- `pct_chg`（涨跌幅百分比）而非 `pct_change`

---

#### 解决方案

重写涨跌幅计算逻辑，兼容多种字段名：

```javascript
const priceChange = computed(() => {
  const info = stockInfo.value
  if (info.price && info.last_close) {
    return info.price - info.last_close
  }
  if (info.close && info.pre_close) {
    return info.close - info.pre_close
  }
  return 0
})

const priceChangePct = computed(() => {
  const info = stockInfo.value
  if (info.pct_chg !== undefined && info.pct_chg !== null) {
    return Number(info.pct_chg)
  }
  // fallback: 手动计算
  const change = priceChange.value
  const base = (info.last_close || info.pre_close || 1)
  return (change / base) * 100
})
```

## 技术要点总结

### 混合模型架构

```
        原始数据
           │
    ┌──────┴──────┐
    │             │
    ▼             ▼
 时序数据      结构化数据
（价格/成交量） （技术指标/财务）
    │             │
    ▼             ▼
  LSTM模型     LightGBM/
（捕捉时序依赖）  CatBoost
    │             │
    └──────┬──────┘
           ▼
     堆叠集成层
           │
           ▼
     最终预测结果
```

### 集成方式

- **Stacking（堆叠集成）：** LSTM 提取特征后与原始特征拼接，输入 LightGBM 进行最终预测
- **优势：** 发挥两类模型的互补优势，提升约 10-15% 的预测准确率

### FAQ-032: Walk-Forward 验证返回 404 Not Found

**发生时间：** 2026-05-20 11:14
**问题类型：** API 路由注册 / 数据验证问题
**影响范围：** Walk-Forward 交叉验证功能

---

#### 问题描述

前端调用 Walk-Forward 验证接口返回 404 错误：
```
POST /api/predict/hybrid/walkforward HTTP/1.1" 404 Not Found
```

但后端日志显示已进入函数：
```
[INFO] [api.predict] 开始 Walk-Forward 验证: 600433
```

---

#### 错误分析

问题出在数据验证阶段，使用了错误的 HTTP 状态码 `404`（资源不存在），且缺乏详细日志：

```python
# 原来的代码
if df.empty or len(df) < req.train_window + req.test_window + 50:
    raise HTTPException(status_code=404, detail="数据不足...")
```

1. **错误码不当**：404 表示"资源不存在"，但这里是"参数不足"，应该用 400
2. **缺乏日志**：未记录实际获取的数据量，难以排查
3. **默认值过大**：train_window=200 + test_window=20 + 缓冲50 = **270条数据**

---

#### 解决方案

**1. 修复后端数据验证逻辑：**

```python
# 添加详细日志
logger.info(f"Walk-Forward 获取到 {len(df)} 条原始数据 (需要 >= {req.train_window + req.test_window + req.seq_length + 10} 条)")

# 使用动态缓冲替代固定50
min_required = req.train_window + req.test_window + req.seq_length + 10

if df.empty or len(df) < min_required:
    raise HTTPException(
        status_code=400,  # 改为 400（参数错误）
        detail=f"数据不足：获取到 {len(df)} 条数据，Walk-Forward 验证至少需要 {min_required} 条。"
               f"建议：扩大日期范围 或 减小训练窗口(train_window 当前={req.train_window})"
    )
```

**2. 降低前端默认参数：**

| 参数 | 修改前 | 修改后 | 说明 |
|------|--------|--------|------|
| `train_window` | 200 | **120** | 降低数据量需求 |
| `test_window` | 20 | **15** | 减少每个窗口消耗 |
| `lstm_epochs` | min(30) | **min(20)** | 加速验证 |

**3. 优化最低数据量计算：**

```
原公式: train_window + test_window + 50
新公式: train_window + test_window + seq_length + 10（动态）

示例（seq_length=20）:
- 原: 200 + 20 + 50 = 270 条
- 新: 120 + 15 + 20 + 10 = 165 条（降低 39%）
```

| 文件路径 | 修改内容 |
|----------|----------|
| `backend/api/predict.py` | 修复数据验证逻辑和日志 |
| `frontend/web/src/pages/Train.vue` | 降低 Walk-Forward 默认参数 |

### FAQ-043: 预测历史"准确率"列始终为空，无回填机制

**发生时间：** 2026-05-20 20:55
**问题类型：** 功能缺失 — 预测后缺少实际价格回填与准确率计算
**影响范围：** 预测历史记录的准确率展示

---

#### 问题描述

5月19日预测的股票记录，到了5月20日应该能拿到实际价格来计算准确率，但"准确率"列始终显示 `-`。所有历史记录的 `accuracy` 字段都是 NULL。

---

#### 错误分析

**根本原因：预测保存时未写 `actual_price/accuracy`，查询时也无回填逻辑。**

| 环节 | 现状 |
|------|------|
| **预测保存** (`predict.py:216-224`, `predict.py:619-629`) | 只写了 code/date/model_name/current_price/predicted_price/confidence，`actual_price` 和 `accuracy` 为 NULL |
| **历史查询** (`get_prediction_history`) | 直接返回 `r.accuracy`（NULL），不做任何回填处理 |

数据库设计中有这两个字段（`database.py:102-103`）：
```python
actual_price = Column(Float, comment='实际价格(后续更新)')
accuracy = Column(Float, comment='准确率(后续计算)')
```

但**从未有代码实现"后续更新/计算"的逻辑**。

---

#### 解决方案

在 `get_prediction_history` 接口中实现自动回填逻辑：

```
1. 查询历史记录
2. 过滤出 date < today 且 actual_price 为 NULL 的记录
3. 批量拉取这些股票的最新收盘价（避免重复请求）
4. 对每条待回填记录：
   a. 用最新收盘价作为 actual_price
   b. 计算：(实际价格 - 当时价格) / 当时价格 → 判断涨跌方向
      - 涨幅 > 0.3% → 实际上涨
      - 跌幅 > 0.3% → 实际下跌  
      - 其他 → 震荡
   c. 对比预测方向 → 命中则 accuracy=100%，否则=0%
5. 持久化到数据库（actual_price + accuracy）
6. 返回带准确率的数据给前端
```

**关键设计决策：**
- 使用 **0.3% 阈值** 区分涨跌（过滤噪音）
- 准确率为**二值**（100% 或 0%），因为单次预测要么对要么错
- 回填只在**查询时触发**，不需要定时任务
