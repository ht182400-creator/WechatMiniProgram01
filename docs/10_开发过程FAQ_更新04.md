### 2.1 股票名称仍显示为数字修复（2026-05-22）

**问题现象：**
股票详情页头部名称显示为股票代码（如 `300433`），而非中文名称（如 `蓝思科技`）。

**根本原因：**
WebSocket 实时行情推送覆盖了从搜索接口获取的中文名称。WebSocket 返回的行情数据中 `name` 字段可能为空或为代码，`watch(quotes[stockCode])` 将其直接赋值给 `stockInfo.name`，覆盖了正确的中文名称。

**修复方式：**

**1. StockDetail.vue — 3级名称获取策略：**

```javascript
// 优先级: realtime → search（宽松匹配）→ getList 兜底
const fetchStockName = async (code) => {
  // 第1级：从实时行情获取
  const res = await stockApi.getRealtime([code])
  if (res?.data?.[0]?.name && /[\u4e00-\u9fa5]/.test(res.data[0].name)) {
    return res.data[0].name
  }
  // 第2级：搜索（宽松匹配：尾部匹配或包含）
  const sr = await stockApi.search(code)
  const match = sr?.data?.find(s => 
    s.code === code || s.code?.endsWith?.(code) || code.endsWith?.(s.code)
  )
  if (match?.name) return match.name
  // 第3级：全量列表兜底
  const list = await stockApi.getList()
  const found = list?.data?.find(s => s.code === code)
  return found?.name || code
}
```

**2. WebSocket watcher 中文保护：**

```javascript
// 仅在当前名称为非中文时才接受 WebSocket 覆盖
watch(() => quotes.value?.[stockCode.value], (data) => {
  if (!data) return
  const currentName = stockInfo.value.name
  const hasChinese = /[\u4e00-\u9fa5]/.test(currentName || '')
  if (!hasChinese && data.name) {
    stockInfo.value.name = data.name
  }
  // 价格数据正常更新...
})
```

**修改文件：** `frontend/web/src/pages/StockDetail.vue`

### 2.2 分钟K线不显示修复（2026-05-22）

**问题现象：**
选择分钟K线周期（1分钟/5分钟/15分钟/30分钟/60分钟）后，图表区域一直显示"K线数据加载中..."，始终无数据渲染。

**错误日志：**
```
AttributeError: 'DataSourceManager' object has no attribute 'get_minute_data'
```

**根本原因：**
`DataSourceManager` 虽然有 `get_daily_data()` 方法，但完全缺少 `get_minute_data()` 方法。前端请求分钟K线时，后端 API 调用 `self.manager.get_minute_data()` 直接抛出 `AttributeError`，导致 500 错误。

此外 `get_depth_data()` 和 `get_transaction_data()` 方法也存在同样缺失。

**修复方式：**

在 `backend/adapters/data_source_manager.py` 新增3个委托方法：

```python
def get_minute_data(self, code: str, period: str = '5') -> pd.DataFrame:

    """
    if not code:
        return pd.DataFrame()
    for name in ['tdx_local', 'tdx_hq', 'akshare']:
        if name in self.adapters:
            try:
                adapter = self.adapters[name]
                if not adapter.connected:
                    adapter.connect()

                    return df
            except Exception as e:
                logger.warning(f"从 {name} 获取分钟K线失败: {e}")
    logger.error(f"所有数据源均无法获取分钟K线: {code} {period}")
    return pd.DataFrame()

def get_depth_data(self, code: str) -> pd.DataFrame:
    """获取十档盘口数据"""
    # 类似的兜底逻辑
    ...

def get_transaction_data(self, code: str, limit: int = 100) -> pd.DataFrame:
    """获取分笔成交数据"""
    # 类似的兜底逻辑
    ...
```

### 2.3 周K/月K显示日线数据修复（2026-05-22）

**问题现象：**
切换到周K或月K周期时，图表数据与日K完全一致，K线数量、涨跌形态均未改变。

**根本原因：**
后端 `/api/stock/chart` 端点（`backend/api/stock.py` 第207行）完全忽略了前端传入的 `chart_type` 参数，无论请求的是 `weekly` 还是 `monthly`，始终调用 `get_daily_data()` 返回日线数据。

**修复方式：**

在 `/stock/chart` 端点中新增周/月线重采样逻辑：

```python
@app.get("/stock/chart")
async def get_chart_data(code: str, chart_type: str = 'daily', ...):
    # 1. 根据周期调整获取原始数据量
    if chart_type == 'weekly':
        data_days = days * 7  # 周线需要7倍原始数据
    elif chart_type == 'monthly':
        data_days = days * 30  # 月线需要30倍原始数据
    else:
        data_days = days
    
    # 2. 获取日线原始数据
    df = manager.get_daily_data(code, ...)
    
    # 3. 周/月线通过 pandas resample 重采样
    if chart_type == 'weekly':
        df = df.resample('W', on='date').agg({
            'open': 'first',
            'high': 'max',
            'low': 'min',
            'close': 'last',
            'volume': 'sum',
            'amount': 'sum'
        }).dropna(subset=['open', 'close'], how='all')
    elif chart_type == 'monthly':
        df = df.resample('ME', on='date').agg({
            'open': 'first',
            'high': 'max',
            'low': 'min',
            'close': 'last',
            'volume': 'sum',
            'amount': 'sum'
        }).dropna(subset=['open', 'close'], how='all')
    
    return df.to_dict(orient='records')
```

**重采样规则说明：**
| 字段 | 聚合方式 | 说明 |
|------|---------|------|
| open | first | 周期第一根K线的开盘价 |
| high | max | 周期内的最高价 |
| low | min | 周期内的最低价 |
| close | last | 周期最后一根K线的收盘价 |
| volume | sum | 周期内成交量总和 |
| amount | sum | 周期内成交额总和 |

**修改文件：** `backend/api/stock.py`

### 2.4 ECharts 崩溃修复（dataZoom越界 + 实例管理）（2026-05-22）

**问题现象：**
控制台连续报错，ECharts 图表白屏或崩溃：

1. `xAxis "4" not found` — dataZoom 配置了不存在的 xAxis 索引
2. `Cannot read properties of null` — chartInstance 为 null 时调用 setOption
3. `setOption should not be called during main process` — 在 Vue 更新周期内同步调用 setOption

**根本原因分析：**

**原因1 — dataZoom 越界索引：**
```javascript
// 旧代码：VOL关闭时 volGridHas = -1
const volGridHas = hasVOL ? 1 : -1
dataZoom: [{
  xAxisIndex: [0, volGridHas, 2, 3, 4]  // 当 volGridHas=-1 时，-1 是无效索引
}]
```
当成交量面板关闭时，`volGridHas` 被设为 `-1`，导致 `xAxisIndex` 数组中包含 `-1` 这个无效索引。ECharts 尝试查找 `xAxis[4]`（序号4的x轴），但实际上只有0-3号x轴存在，触发 `xAxis "4" not found` 错误。

**原因2 — setOption 时序错误：**
`setOption` 在 Vue 的响应式更新周期（`watch` / `computed` 回调）内被同步调用，ECharts 不允许在渲染主过程中修改图表配置。需要使用 `requestAnimationFrame` 将调用延迟到下一个浏览器帧。

**修复方式：**

**1. dataZoom 索引动态计算（HQChartKline.vue）：**

```javascript
// 修复前：硬编码 xAxisIndex，包含可能越界的索引
const volGridHas = hasVOL ? 1 : -1
xAxisIndex: [0, volGridHas, 2, 3, 4]

// 修复后：动态收集实际存在的 grid 索引
const gridIndexes = [0] // 主图 grid 始终存在
if (hasVOL) gridIndexes.push(1)  // VOL画布存在才加
if (hasKDJ) gridIndexes.push(2)  // KDJ画布存在才加
if (hasMACD) gridIndexes.push(3) // MACD画布存在才加
if (hasRSI) gridIndexes.push(4)  // RSI画布存在才加

// dataZoom 配置
dataZoom: [{
  xAxisIndex: gridIndexes,  // 仅包含有效索引
  ...
}]
```

**2. ECharts 实例生命周期管理：**

```javascript
// 每次 rebuild 前先销毁旧实例
if (chartInstance) {
  try { chartInstance.dispose() } catch (_) {}
  chartInstance = null
}

// 创建新实例
chartInstance = echarts.init(chartRef.value)

// 使用 requestAnimationFrame 脱离 Vue 更新周期
requestAnimationFrame(() => {
  try {
    chartInstance.setOption(option, true)
  } catch (e) {
    console.error('ECharts setOption 失败:', e)
    // 失败时 dispose 损坏实例，下次 rebuild 重新创建
    try { chartInstance?.dispose() } catch (_) {}
    chartInstance = null
  }
})
```

**修改文件：** `frontend/web/src/components/HQChartKline.vue`

## 五、Bug 修复：分钟K线周期偏移（2026-05-22）

### 5.1 问题现象

K线图选择不同分钟周期时，显示的数据与标签不匹配：

| 用户选择的周期 | 实际显示的周期 | 偏移量 |
|---------------|--------------|--------|
| 1 分钟 | 空数据（或5分） | - |
| 5 分钟 | **15 分钟** | 偏移1级 |
| 15 分钟 | **30 分钟** | 偏移1级 |
| 30 分钟 | **60 分钟** | 偏移1级 |
| 60 分钟 | **日线** | 偏移1级 |

### 5.2 根因分析

mootdx 库内部 `FREQUENCY` 列表定义为：

```python
# mootdx/quotes/frequency.py
FREQUENCY = ['5m', '15m', '30m', '1h', 'day', 'week', 'mon', '1m', ...]
#   index:     0     1      2      3    4      5       6     7
```

旧代码 `tdx_hq_adapter.py` 的 FREQ_MAP 按错误假设 `0=1m, 1=5m, 2=15m...` 编写：

```python
# ❌ 错误映射
FREQ_MAP = { 1:0, 5:1, 15:2, 30:3, 60:4 }
# 实际效果: 5分→index1→'15m'(15分), 60分→index4→'day'(日线)
```

### 5.3 修复方式

**文件1：`backend/adapters/tdx_hq_adapter.py`**

重写 FREQ_MAP 为正确的 FREQUENCY 索引映射：

```python
# ✅ 正确映射（基于 FREQUENCY 列表实际索引）
FREQ_MAP = {
    1: 7,    # 1分钟 → FREQUENCY[7] = '1m'
    5: 0,    # 5分钟 → FREQUENCY[0] = '5m'
    15: 1,   # 15分钟 → FREQUENCY[1] = '15m'
    30: 2,   # 30分钟 → FREQUENCY[2] = '30m'
    60: 3,   # 60分钟 → FREQUENCY[3] = '1h'
}
```

**文件2：`backend/adapters/tdx_local_adapter.py`**

重构 `get_minute_data()` 方法：

- 1分/5分：直接从本地文件读取（`reader.minute()` 仅支持这两种）
- 15分/30分/60分：先获取1分钟原始数据，再用 pandas `resample('15min')` 聚合
- 新增 `_get_raw_minute_data()` 和 `_resample_minutes()` 辅助方法
- 修正手动解析文件的后缀映射（`{1:'.lc', 5:'.lc1'}`）

## 八、Bug 修复：WinError 64 噪声日志静默（2026-05-22）

### 8.1 问题现象

后端控制台间歇性输出以下冗余错误日志：

```
[ERROR] [asyncio] Task exception was never retrieved
OSError: [WinError 64] 指定的网络名不再可用。

[ERROR] [asyncio] Accept failed on a socket
OSError: [WinError 64] 指定的网络名不再可用。
```

### 8.2 根因分析

`WinError 64` (`ERROR_NETNAME_DELETED`) 是 Windows IOCP (ProactorEventLoop) 下的已知行为。

**触发场景（均属正常客户端行为）：**
- 浏览器关闭标签页 / 刷新页面 → HTTP 请求中断
- WebSocket 客户端断开连接
- 网络切换或 VPN 断开

**为什么会报错：**
asyncio 的 `ProactorEventLoop.accept_coro` 在处理 TCP accept 时，如果客户端在握手完成前断开，Windows IOCP 会返回 `ERROR_NETNAME_DELETED (64)`。Python 的 `asyncio` 库没有正确静默这个异常，导致两条日志输出：
1. `Task exception was never retrieved` — Future 异步任务未捕获异常
2. `Accept failed on a socket` — uvicorn 内部的 accept 循环日志

**影响评估：** 对系统功能无任何影响，纯属噪声日志，干扰排查真正的问题。

### 8.3 修复方式

**文件：`backend/main.py`**

在 `__name__ == "__main__"` 入口处注册 `logging.Filter`，挂载到 `asyncio` 和 `uvicorn.error` 两个 logger，自动过滤包含 WinError 64 的日志记录。

```python
import logging as _logging

class WinError64Filter(_logging.Filter):
    def filter(self, record):
        msg = record.getMessage()
        if '指定的网络名不再可用' in msg:
            return False
        if 'WinError 64' in msg:
            return False
        return True

_logging.getLogger('asyncio').addFilter(WinError64Filter())
_logging.getLogger('uvicorn.error').addFilter(WinError64Filter())
```

**为什么使用 logging Filter 而非事件循环异常处理器：**
1. `uvicorn.run(reload=True)` 会创建子进程并重建事件循环，直接在 `asyncio.new_event_loop()` 上设置异常处理器在 reload 模式下会丢失
2. logging Filter 在 Python 进程级别注册，无论 uvicorn 如何管理子进程，都会生效
3. 对原有日志体系零侵入，不影响真正需要关注的错误

## 十二、WS 推送覆盖中文名称问题（2026-05-22）

### 12.1 问题现象

1. 首次加载页面 → `fixStockNames()` 修复成功，自选股显示中文名（浦发银行、平安银行等）
2. WebSocket 推送实时行情后 → 名称瞬间变回数字（491、503、488、505...）
3. 反复刷新可复现

### 12.2 根因分析

**数据流追踪：**

```
后端 websocket_manager.py:223
  'name': row.get('name', code)   ← 若 DataFrame 的 name 列为空/异常，fallback 到纯数字代码
       ↓ WS 推送
前端 Home.vue (修复前)
  name: q.name || stock.name     ← q.name 是数字 "491"（非空字符串，truthy）→ 直接覆盖中文名！
```

**关键错误：** `q.name || stock.name` 使用 `||` 运算符，只要 `q.name` 不是 `null/undefined/""` 就会采用。但后端推送的 `name` 可能是：
- 纯数字代码（如 `"491"`）
- 内部 ID
- 其他非中文字符串

这些都是 **truthy 值**，导致已修复的中文名被反复篡改。

### 12.3 修复方案

在 `watch(quotes)` 回调中新增 `safeName()` 函数：

```javascript
/**
 * 安全合并股票名称 — 仅接受包含中文的名称，防止 WS 推送的数字/代码覆盖本地中文名
 */
const safeName = (wsName, localName) => {
  if (wsName && /[\u4e00-\u9fa5]/.test(wsName)) return wsName
  return localName
}
```

使用方式（两处调用点均替换）：
```javascript
// 自选股列表更新
return { ...stock, name: safeName(q.name, stock.name), ... }
// 当前选中股票更新  
name: safeName(q.name, selectedStock.value.name)
```

| 调用点 | 行号 | 说明 |
|--------|------|------|
| watchlist map | ~449 | 自选股侧边栏 + K线标题 |
| selectedStock | ~461 | 主图表头部信息栏 |

  代码 `000001` 既可以是平安银行（股票），也可以是上证指数（指数），必须用不同接口查询。原代码统一走 `quotes()` 导致混淆。

- **修复方案**：在 `data_source_manager.get_realtime_quote()` 中增加**智能路由**：

  ```python
  # 定义已知指数代码集合
  INDEX_CODES = {'000001', '399001', '399006', '000300', ...}
  
  def strip_prefix(c):
      return re.sub(r'^(sh|sz)', '', c)

  index_codes = [strip_prefix(c) for c in codes if strip_prefix(c) in INDEX_CODES]
  
  # 指数 → get_index_realtime() → mootdx index()
  # 股票 → get_realtime_quote() → mootdx quotes()
  # 最后 pd.concat 合并返回
  ```

## 十八、WebSocket JSON 解析错误 — NaN 问题（2026-05-22）

### 18.1 问题现象

前端控制台反复报错：

```
[WS] 消息解析失败: SyntaxError: Unexpected token 'N', ..."0,"price":NaN,"open""... is not valid JSON
```

### 18.2 根因分析

**JSON 规范不支持 `NaN`、`Infinity`、`-Infinity`，只支持 `null`。**

后端 `websocket_manager.py:224-229` 使用 `float()` 转换 DataFrame 数值时：
```python
'price': float(row.get('close', row.get('price', 0)))
```

如果 DataFrame 中某列为空值（pandas `NaN`）：
- Python: `float(numpy.nan)` → `nan`
- `json.dumps(nan)` → 输出 `NaN`（非法 JSON！）
- 前端 `JSON.parse('NaN')` → `SyntaxError`

同样，涨跌幅计算的中间结果也可能产生 NaN：
```python
quote['data']['pct_change'] = (change / 0) * 100  # division by zero → NaN
```

### 18.3 修复方案

新增 `safe_float()` 函数包裹所有数值转换，NaN/Infinity 替换为 0：

```python
import math

def safe_float(val, default=0.0):
    """将值转为 float，NaN/Inf 替换为 default"""
    try:
        v = float(val)
        if math.isnan(v) or math.isinf(v):
            return float(default)
        return v
    except (ValueError, TypeError):
        return float(default)
```

应用范围：
- `price`, `open`, `high`, `low`, `volume`, `amount` → `safe_float()`
- `change`（涨跌额）→ 计算后额外检查 `math.isnan()`
- `pct_change`（涨跌幅）→ 计算后额外检查 `math.isnan()`

### 18.4 修改文件

| 文件 | 改动 |
|------|------|
| `backend/api/websocket_manager.py` | `_push_quotes()` 新增 `safe_float()`，所有数值字段 NaN 防护 |

发现 **3 个叠加问题**：

#### 问题 A：`000001` 被误判为指数代码

**影响范围**：`000001`(平安银行)、`399001` 等与指数同名的股票代码均受影响。

### 20.2 后端：NoCacheMiddleware — 禁用 API 响应缓存

**问题现象：**
浏览器可能缓存 API 响应，导致切换页面或刷新时显示旧数据。

**修复方式：**

在 `backend/main.py` 新增 `NoCacheMiddleware`：

```python
class NoCacheMiddleware(BaseHTTPMiddleware):
    """禁用 API 响应缓存，确保浏览器始终获取最新数据"""

    async def dispatch(self, request: Request, call_next):
        response: Response = await call_next(request)
        if request.url.path.startswith('/api/'):
            response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, max-age=0'
            response.headers['Pragma'] = 'no-cache'
            response.headers['Expires'] = '0'
        return response

app.add_middleware(NoCacheMiddleware)
```

### 20.6 后端：股票列表中文名称质量校验

def _check_chinese_names(self, df: pd.DataFrame) -> bool:
    """验证 DataFrame 中 name 列是否包含有效中文名称"""
    if 'name' not in df.columns:
        return False
    import re
    chinese_count = df['name'].apply(
        lambda x: bool(re.search(r'[\u4e00-\u9fa5]', str(x)))

### 20.7 后端：Parquet 缓存引擎检测 + Pickle 降级

class ParquetCache:

        try:

            pd.read_parquet(test_path)

        except Exception:
            return False

    def _get_cache_path(self, key: str) -> Path:

### 20.10 前端：API 层超时 + 防缓存增强

**1. 请求超时延长至 60s（`api/index.js`）：**

```javascript
const apiClient = axios.create({
  baseURL: '/api',
  timeout: 60000,  // 60s（akshare/baostock 首次抓取较慢）
})
```

**2. 请求拦截器自动添加防缓存时间戳：**

```javascript
apiClient.interceptors.request.use(config => {
  if (config.method === 'get') {
    config.params = config.params || {}
    config.params._t = Date.now()
  }
  return config
})
```

**双重防缓存：** 请求端自动加 `_t` 参数 + 后端 `NoCacheMiddleware` 设置响应头，确保数据实时性。

**修改文件：** `frontend/web/src/api/index.js`

### 20.13 前端：core-js Polyfill 解决 HQChart 兼容性

**问题现象：**
Vite 默认"现代浏览器优先"策略可能导致 HQChart（依赖 babel-runtime）在旧浏览器中缺少核心 API。

**修复方式：**
在 `main.js` 入口最顶部添加 polyfill：

```javascript
/** ★ Polyfill: Vite 现代浏览器优先策略可能导致 HQChart 依赖的核心 API 缺失 */
import 'core-js/stable'
import 'regenerator-runtime/runtime'
```

**依赖安装：**

```json
// package.json 新增
"core-js": "^3.x",
"regenerator-runtime": "^0.x"
```

**修改文件：**
- `frontend/web/src/main.js`
- `frontend/web/package.json`

### 20.14 前端：Vite 依赖预构建优化

**配置（`vite.config.js`）：**

```javascript
optimizeDeps: {
  include: ['hqchart', 'jquery']
}
```

**原因：** HQChart 是 CommonJS 依赖，内部引用 `babel-runtime` 和 `jquery`。Vite 开发模式下需要预构建这些 CommonJS 模块才能正确运行。不加此配置可能导致开发环境白屏或 K 线图组件加载失败。

**修改文件：** `frontend/web/vite.config.js`
